## Context

В существующем корпусе `source_authority` содержит значения разных классов: уровни власти, признак гражданского источника и имена конкретных операторов. Одновременно уже существуют более точные поля `jurisdiction_level`, `legal_significance` и типизированные связи `published_by` / `operated_by`. Экспорт версии 1 допускает добавление nullable-полей без повышения версии.

## Goals / Non-Goals

- Goals: однозначная provenance-модель; отсутствие потери legacy-данных; сопоставимое качество между семействами; достижимость массовых импортов из хабов.
- Non-Goals: удаление `source_authority` в этой версии; обязательное ручное редактирование каждой импортной карточки; вывод доверия к данным только по уровню органа.

## Decisions

- Decision: уровень публикации хранится в `jurisdiction_level`, юридический статус/тип источника — в `legal_significance`, издатель и оператор — направленными `relationships` на канонические карточки организаций.
- Decision: `source_authority` становится legacy-полем. Новые карточки SHALL NOT использовать его как единственный provenance-признак; экспорт временно сохраняет значение для миграции и диагностики.
- Decision: nullable-добавление `source_authority` и полей качества в индекс остаётся в `schema_version: 1`; удаление или переименование откладывается до отдельной версии схемы.
- Decision: массовые семейства получают значения из проверенных правил импорта только там, где правило действительно одинаково для семейства. Неизвестное значение сериализуется как `null`, а не угадывается.
- Decision: читательская достижимость обеспечивается хабами семейства. Не требуется вручную добавлять ссылку на каждую импортную карточку из тематической статьи, если канонический хаб перечисляет или детерминированно группирует всё семейство.

## Risks / Trade-offs

- Массовая миграция может создать ложную однородность → значения импортируются только из документированного правила, исключения помечаются для редакционного прохода.
- Временное сохранение legacy-поля увеличивает контракт → поле явно помечается deprecated и не используется как единственный фильтр.
- Хаб на сотни карточек может быть неудобен → допускается иерархия индексных страниц, если каждая карточка достижима и группы понятны читателю.

## Migration Plan

1. Архивировать `expand-structured-knowledge-layer` и принять его спеки как канонические.
2. Добавить nullable-ключи экспорта и отчёт текущего покрытия без изменения карточек.
3. Обновить шаблон, справочник и импортёры для новых карточек.
4. Мигрировать curated-семейства, затем массовые импорты по документированным правилам.
5. Добавить хабы и включить проверку достижимости.
6. После отдельного периода совместимости оценить proposal для schema v2 и удаления `source_authority`.

## Open Questions
