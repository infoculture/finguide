# Capability: reporting-landscape

Ландшафт бюджетной отчётности: единый вход в раздел, связи между формами, задачи аналитика, история редакций форм и связь с примерами на данных. Источник целей: [_dev/content_plan_20260514.md](../../../_dev/content_plan_20260514.md), разд. 5.3, 6, 7.

## Requirement: Reporting overview entry point

Проект SHALL обеспечивать **одну** каноническую точку входа «как устроена бюджетная отчётность» для читателя сайта:

- **Вариант A (предпочтительный):** отдельная страница [`wiki/reporting/reporting-overview.md`](../../../wiki/reporting/reporting-overview.md) (или согласованное имя файла со slug `/reporting/<slug>`) с полным обзором; [`wiki/reporting/README.md`](../../../wiki/reporting/README.md) остаётся хабом и **должен** ссылаться на обзор в первых разделах.
- **Вариант B:** расширить [`wiki/reporting/README.md`](../../../wiki/reporting/README.md) до полноценного обзора; тогда отдельный `reporting-overview.md` не создаётся, и это решение MUST быть отражено в коммите и в [`sidebars.ts`](../../../sidebars.ts), если обзор должен быть в сайдбаре.

Обзор MUST описывать: исполнение бюджета, консолидированные бюджеты субъектов, месячную динамику (где применимо), источники финансирования дефицита, долг, учреждения, межбюджетные трансферты — **в терминах форм и разделов корпуса**, со ссылками на существующие карточки [`wiki/reporting/`](../../../wiki/reporting/).

### Scenario: Reader lands from README

- **GIVEN** читатель открывает README раздела «Отчётность»
- **WHEN** он следует ссылке на обзор (или читает README как обзор в варианте B)
- **THEN** он понимает, какую форму открыть для типовой задачи без перебора всех 22 карточек

## Requirement: Form relationship diagram

Обзор или связанная страница MUST содержать диаграмму связей форм отчётности в формате **Mermaid** (например `flowchart` или `graph`), охватывающую согласованный набор карточек форм, перечисленный в обзоре.

- Экспорт в PNG MAY добавляться для удобства печати; если PNG хранится в репозитории, рядом MUST быть исходная Mermaid-диаграмма в markdown, чтобы её можно было обновлять.
- При добавлении новой карточки формы из списка «затронутых связей» диаграмма SHOULD обновляться в том же редакционном пакете или в последующем PR с явной ссылкой на задачу.

### Scenario: Analyst navigates between forms

- **GIVEN** пользователь изучает диаграмму связей
- **WHEN** он переходит по ссылкам на карточки форм
- **THEN** он находит описание назначения формы и раздел «Где встречается в данных»

## Requirement: Analytical task to form to source matrix

Проект SHALL содержать таблицу **«аналитическая задача → форма (или группа форм) → источник данных / ИС → how-to»** на странице обзора или на отдельной странице (аналог `forms-relationship-map.md`).

Минимальные строки MUST покрывать сценарии:

- исполнение федерального бюджета по КБК и следы в открытых данных / Казначействе;
- консолидированный бюджет субъекта и региональные источники;
- связка с электронными отчётами и порталами, где это уже отражено в корпусе.

### Scenario: Task row is actionable

- **GIVEN** строка таблицы описывает задачу
- **WHEN** читатель переходит по каждой ссылке в строке
- **THEN** он попадает на опубликованную страницу корпуса с осмысленным продолжением маршрута

## Requirement: Form revision history in cards

Карточки отчётности в [`wiki/reporting/`](../../../wiki/reporting/) SHOULD содержать явное описание истории изменений формы (год введения, существенные перестройки, ссылка на нормативку) в теле карточки и/или во frontmatter, если в [`wiki/reporting/reporting-card-template.md`](../../../wiki/reporting/reporting-card-template.md) введены согласованные поля.

Новые и существенно обновляемые карточки MUST приводиться к шаблону по мере правок.

### Scenario: Retrospective user sees form cut-off

- **GIVEN** пользователь строит ряд за прошлые годы
- **WHEN** он открывает карточку формы
- **THEN** он видит, применима ли форма к выбранному году или куда смотреть за предыдущей редакцией

## Requirement: Examples tied to exports

Обзор или карточки форм SHOULD ссылаться на how-to или короткие примеры на реальных выгрузках, удовлетворяющие [`howto-playbook`](../howto-playbook/spec.md) и [`reproducible-analytics`](../reproducible-analytics/spec.md).

### Scenario: Example link from reporting card

- **GIVEN** карточка формы описывает машиночитаемый след
- **WHEN** читатель переходит по ссылке на how-to
- **THEN** how-to содержит разделы «Входные данные», «Воспроизводимый пример» и «Проверка результата»
