# Capability: budget-classification-temporal

Временно́е измерение бюджетной классификации (КБК, КВР, КОСГУ, целевые статьи): хронология, соответствия между годами, практические расшифровки и связь с отчётностью и источниками данных. Источник целей: [_dev/content_plan_20260514.md](../../../_dev/content_plan_20260514.md), разд. 5.1, 6, 7.

## Requirement: Classification change chronicle

Проект SHALL содержать страницу-обзор изменений бюджетной классификации по годам под [`wiki/budget-classification/`](../../../wiki/budget-classification/) (содержательный аналог предложенного `classification-changes-by-year.md`).

Страница MUST:

- связывать периоды применимости редакций с каноническими карточками НПА в [`wiki/legal/`](../../../wiki/legal/) (приказы Минфина и смежные акты), где они есть в корпусе;
- отделять описание «что изменилось» от руководства «как читать старые выгрузки и отчётность»;
- содержать перекрёстные ссылки на [`wiki/glossary/`](../../../wiki/glossary/) и [`wiki/budget-classification/overview.md`](../../../wiki/budget-classification/overview.md) (или актуальный обзор раздела).

### Scenario: Reader finds applicable year

- **GIVEN** аналитик открывает страницу хронологии классификации
- **WHEN** он ищет календарный год анализа
- **THEN** он находит применимую редакцию, ссылку на нормативную базу и пояснение ограничений для ретроспективы

### Scenario: Legal anchor exists or is explicit

- **GIVEN** для года указано изменение классификации
- **WHEN** в корпусе есть карточка соответствующего акта
- **THEN** страница содержит ссылку на `/legal/<slug>` с пояснением строки

## Requirement: Machine-readable correspondence tables

Для ключевых переходов классификации (КБК, КВР, КОСГУ, целевые статьи) проект SHALL поддерживать **парность**: машиночитаемый артефакт и человекочитаемое резюме.

- Каноническое расположение CSV: каталог **`wiki/budget-classification/assets/`** (создавать по мере появления файлов). Альтернативные корневые каталоги (`_data/` и т.п.) MUST NOT использоваться для этих таблиц без обновления данной спеки.
- Каждый CSV SHOULD иметь сопроводительный абзац или подраздел на странице хронологии или на связанной странице классификации с описанием покрытия и ограничений.

### Scenario: Analyst downloads mapping

- **GIVEN** опубликован переход между редакциями
- **WHEN** пользователь открывает репозиторий или собранный сайт
- **THEN** он может открыть CSV из `wiki/budget-classification/assets/` и понять столбцы из сопроводительного текста

## Requirement: Worked KBK examples

Проект SHALL содержать не менее **пяти** и рекомендуется **десять** разобранных примеров кодов (КБК или эквивалентная декомпозиция с явной ссылкой на структуру кода), охватывающих как минимум:

- доходы бюджета;
- расходы бюджета;
- межбюджетные трансферты;
- источники финансирования дефицита.

Примеры могут размещаться на одной или нескольких страницах [`wiki/budget-classification/`](../../../wiki/budget-classification/) и/или в how-to (например расшифровка КБК), с **конкретными** ссылками на [`wiki/glossary/`](../../../wiki/glossary/), [`wiki/legal/`](../../../wiki/legal/), [`wiki/reporting/`](../../../wiki/reporting/), [`wiki/howto/`](../../../wiki/howto/).

### Scenario: Novice follows one full example

- **GIVEN** читатель открывает пример расшифровки
- **WHEN** он проходит по ссылкам «термин → форма → источник»
- **THEN** он получает связный маршрут без заглушек вроде единственной ссылки на корень глоссария

## Requirement: Code to reporting to source map

Материалы по бюджетной классификации SHOULD содержать явную карту минимум для согласованного набора кодов: **код (или класс кодов) → форма отчётности → источник данных → how-to**.

Ссылки MUST использовать канонические slug вида `/reporting/<slug>`, `/data-sources/<подраздел>/<slug>`, `/howto/...` в соответствии с [`AGENTS.md`](../../../AGENTS.md).

### Scenario: Map resolves to open data

- **GIVEN** описан класс расходных КБК для задачи исполнения
- **WHEN** читатель ищет машиночитаемый след
- **THEN** карта ведёт к карточке источника данных и/или how-to с примером доступа

## Requirement: Retrospective applicability markers

Страницы и разделы, где классификация зависит от года или редакции акта, MUST содержать явный маркер периода применимости (год, финансовый год, «до/после») в теле страницы или во frontmatter, если редакция завела согласованное поле.

### Scenario: Stale classification warning

- **GIVEN** материал относится только к прошлым годам
- **WHEN** читатель открывает страницу
- **THEN** он видит предупреждение или дату актуальности, достаточные, чтобы не перенести коды на текущий год без проверки
