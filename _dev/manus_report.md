# Предложения по улучшению репозитория `infoculture/finguide`

**Автор:** Manus AI  
**Дата:** 11 мая 2026  
**Объект анализа:** репозиторий [`infoculture/finguide`](https://github.com/infoculture/finguide)  
**Статус параллельного исследования:** использованы завершённые результаты по 12 из 13 пакетов; незавершённый пакет `legal` был пропущен по просьбе пользователя.

## 1. Краткое резюме

Репозиторий представляет собой Docusaurus-базу знаний о государственных финансах России, ориентированную одновременно на читателей, редакторов и ИИ-ассистентов. Уже сейчас проект имеет сильную основу: содержательная доменная структура, отдельные правила для редакторов и агентов в `AGENTS.md`, строгая сборка Docusaurus, единый каталог `wiki/`, развитая система разделов и большое количество атомарных Markdown-страниц. По локальному аудиту в репозитории найдено **338 Markdown-файлов**, все они содержат YAML-frontmatter, а относительные внутренние Markdown-ссылки не дали очевидных битых ссылок при простой файловой проверке.[1] [2]

Главная зона развития состоит не в полной перестройке, а в доведении базы знаний до состояния **учебного и RAG-ready корпуса**: унифицировать метаданные, сделать маршруты пользователя более явными, усилить цитирование первоисточников, добавить машинно-читаемые индексы и улучшить обзорные страницы, которые сейчас местами остаются короткими навигационными заглушками.

## 3. Сильные стороны репозитория

Репозиторий уже оформлен как **каноническая база знаний**, а не как набор случайных заметок. В `README.md` описана структура проекта, быстрый старт, команды сборки и деплоя, а также даны ссылки на ключевые редакционные документы. Использование Docusaurus 3, `sidebars.ts` и строгих проверок ссылок создаёт техническую основу для устойчивой документации.[1] [3]

Особенно ценно наличие `AGENTS.md`: этот файл задаёт правила для редакторов и ИИ-ассистентов, требует перекрёстных ссылок и описывает шаблоны карточек. Это редкий и сильный элемент, потому что он превращает репозиторий в управляемый корпус знаний, пригодный не только для чтения человеком, но и для полуавтоматического пополнения.

Структура `sidebars.ts` уже отражает ясную предметную модель: введение, глоссарий, организации, информационные системы, бюджетная система, бюджетная классификация, источники данных, инструкции, нормативная база, отчётность и справочники. Такая декомпозиция хорошо соответствует пользовательским сценариям: «понять термин», «найти организацию», «выбрать источник данных», «разобраться в отчётности», «выполнить практическую задачу».[3]

## 4. Основные проблемы и риски

Главный системный риск состоит в том, что формальная структура уже богата, но **не все уровни навигации одинаково содержательны**. Некоторые README-страницы разделов и подразделов слишком короткие и выполняют скорее роль технических узлов, чем обучающих входных точек. Для человека это снижает понятность маршрута, а для ИИ-ассистента создаёт проблему: при попадании в такой документ модель получает мало контекста для ответа.

Второй риск связан с **неполной унификацией метаданных**. Почти все страницы имеют `tags` и `last_updated`, но поле `description` присутствует только у 62 файлов из 338. Для Docusaurus-поиска, RAG-индекса и генерации карточек результатов именно `description` часто становится кратким, контролируемым резюме страницы. Его отсутствие повышает вероятность нерелевантных сниппетов и ухудшает ранжирование.

Третий риск — **разная пригодность страниц для чанкинга**. В корпусе есть 41 длинный файл, включая `wiki/reference/principles.md`, `wiki/intro/wiki-filling-plan.md`, отдельные источники данных, глоссарные статьи и карточки информационных систем. Длинные страницы могут быть полезны человеку, но для RAG они требуют явных заголовков, кратких выводов, таблиц «что это / где искать / как использовать» и стабильных якорей.

Четвёртый риск касается **цитируемости и верифицируемости**. В предметной области государственных финансов важны первоисточники: нормативные акты, официальные порталы, документы Казначейства, Минфина, Росстата и других ведомств. Если карточка объясняет понятие или источник данных без явного блока «Источники и первоисточники», ИИ-ассистенту сложнее отделять авторское объяснение от проверяемого факта.

## 5. Рекомендации по содержанию

### 5.1. Ввести обязательный расширенный frontmatter

Следует закрепить единый набор обязательных и рекомендуемых полей для разных типов страниц. Сейчас наличие frontmatter — сильная сторона, но его нужно сделать более информативным.

| Тип поля | Рекомендация | Зачем это нужно |
|---|---|---|
| `title` | Оставить обязательным. | Человеческая навигация и заголовки документов. |
| `description` | Сделать обязательным для всех неслужебных страниц. | Сниппеты, поиск, RAG-контекст, карточки выдачи. |
| `tags` | Ограничить контролируемым словарём из `reference/tags-taxonomy.md`. | Снижение хаоса в тегах и улучшение фильтрации. |
| `last_updated` | Оставить обязательным и проверять автоматически. | Контроль устаревания. |
| `content_type` | Добавить значения: `concept`, `organization`, `data_source`, `howto`, `legal`, `reporting`, `reference`. | Машинная маршрутизация и сбор датасетов для ИИ. |
| `source_authority` | Добавить для фактических страниц: `official`, `community`, `mixed`, `secondary`. | Оценка доверия при генерации ответов. |
| `rag_priority` | Добавить шкалу `high/medium/low`. | Отбор документов для индексации и обучения. |

### 5.2. Превратить README-разделы в полноценные навигационные хабы

Короткие README-страницы, например в `budget-classification`, `howto` и подразделах `data-sources`, следует расширить. Хороший README должен отвечать на три вопроса: **что есть в разделе, кому он нужен и с чего начать**. Для каждого раздела стоит добавить блок «Типовые вопросы», например: «Где найти исполнение бюджета субъекта РФ?», «Чем КОСГУ отличается от КВР?», «Какая форма отчётности показывает расходы учреждения?».

### 5.3. Добавить единый блок “Как использовать эту страницу”

Для ключевых страниц следует добавить небольшой блок, объясняющий сценарий использования. Например, карточка источника данных может содержать поля «когда использовать», «какие сущности покрывает», «типичные ограничения», «связанные информационные системы», «пример запроса пользователя». Это особенно важно для ИИ-ассистента, который должен выбирать источник под задачу пользователя, а не просто пересказывать описание.

### 5.4. Усилить блоки первоисточников и датировки

Для страниц о нормативных актах, отчётности, источниках данных и информационных системах нужно ввести стандартный блок **«Проверяемые источники»**. В нём должны быть официальные URL, дата обращения или дата актуализации, а для нормативных документов — реквизиты акта и ссылка на действующую редакцию. Это снизит риск галлюцинаций и поможет ассистенту корректно оговаривать актуальность.

### 5.5. Разделить или структурировать длинные страницы

Длинные документы не обязательно дробить автоматически. Лучше применять правило: если страница длиннее условного порога и содержит несколько самостоятельных вопросов, она должна получить либо дочерние страницы, либо устойчивые разделы с якорями, краткое резюме в начале и таблицу связей. Приоритетные кандидаты по аудиту: `reference/principles.md`, `intro/wiki-filling-plan.md`, крупные карточки источников данных, некоторые глоссарные статьи и карточки информационных систем.

## 6. Рекомендации по навигации

Навигация в `sidebars.ts` уже достаточно зрелая, но её можно усилить за счёт сценарных маршрутов поверх предметной структуры. Сейчас сайдбар отвечает на вопрос «где находится раздел», но пользователь часто приходит с вопросом «как решить задачу». Поэтому рекомендуется добавить отдельные страницы-маршруты.

| Маршрут | Где разместить | Содержание |
|---|---|---|
| «Найти данные по федеральному бюджету» | `intro/data-map.md` или `howto/access/` | Термины → организации → ИС → источники → формы отчётности. |
| «Проанализировать региональный бюджет» | `howto/analysis/` | Последовательность источников, классификаций и типовых ошибок. |
| «Понять расходы учреждения» | `reporting/` + `howto/analysis/` | ПФХД, отчётность, субсидии, госзадание, КОСГУ. |
| «Разобраться в закупках и контрактах» | `howto/access/` или `data-sources/` | 44-ФЗ, 223-ФЗ, ЕИС, контракты, отчётность. |
| «Выбрать источник данных для вопроса» | `intro/data-map.md` | Таблица вопрос → сущность → источник → ограничения. |

Второе улучшение — сделать **индексные страницы по сущностям**. Например, отдельные таблицы «Организация → информационные системы → источники данных → связанные страницы», «Термин → классификация → отчётность → источники», «Нормативный акт → регулируемые данные → практические инструкции». Такие таблицы одновременно улучшат навигацию человека и дадут ИИ-ассистенту готовый граф переходов.

Третье улучшение — ввести единый формат ссылок «см. также» в конце страниц. Для RAG это важно, потому что связанные документы не всегда попадают в один контекстный фрагмент. Явный блок связей поможет индексатору строить граф знаний.

## 7. Пригодность материалов для обучения ИИ-ассистента и RAG

В текущем виде репозиторий уже пригоден для первичного RAG-индекса: страницы атомарны, структура каталогов предметная, frontmatter есть у всех Markdown-файлов, а значительная часть навигации задана явно. Однако для использования как **обучающий корпус** требуется дополнительная нормализация. Обучение ассистента требует не только текста, но и устойчивых пар «вопрос → релевантная страница → проверяемый ответ → источник». Сейчас эта структура скорее подразумевается, чем оформлена явно.

| Критерий | Текущая оценка | Что улучшить |
|---|---|---|
| Атомарность | Хорошая, но неоднородная. | Разметить длинные страницы и усилить короткие README. |
| Машиночитаемость | Хорошая за счёт Markdown и frontmatter. | Добавить обязательные `description`, `content_type`, `source_authority`, `rag_priority`. |
| Цитируемость | Средняя. | Ввести стандартный блок первоисточников. |
| Навигационный граф | Хороший в сайдбаре, частично в ссылках. | Добавить сценарные маршруты и таблицы связей. |
| Риск галлюцинаций | Умеренный. | Разделять факты, интерпретации, ограничения и дату актуальности. |
| Подготовка к чанкингу | Средняя. | Добавить резюме, стабильные якоря, секции «ключевые факты» и «ограничения». |

Для ассистента особенно полезны будут дополнительные файлы-индексы в машинно-читаемом формате. Например, можно генерировать `knowledge-index.jsonl`, где каждая строка описывает страницу: путь, заголовок, описание, тип, теги, связанные страницы, официальные источники и рекомендуемые вопросы. Это не заменит Markdown, но сделает корпус пригодным для автоматического отбора и тестирования.

## 9. Quick wins

В первую очередь стоит добавить `description` ко всем страницам, где его нет. Это небольшая редакционная задача с высоким эффектом для поиска, SEO, Docusaurus-страниц и RAG-индекса.

Второй быстрый шаг — расширить короткие README-разделы до формата «назначение → кому полезно → ключевые страницы → типовые вопросы → см. также». Это сразу улучшит навигацию и снизит порог входа.

Третий шаг — добавить в `AGENTS.md` проверочный чек-лист для PR: наличие `description`, актуального `last_updated`, ссылок на первоисточники, блока «см. также», связей с глоссарием и хотя бы одного пользовательского сценария.

Четвёртый шаг — добавить в `package.json` отдельную проверку качества Markdown-корпуса: валидность frontmatter, обязательные поля, допустимые теги, отсутствие пустых README и контроль длины страниц.

## 10. Более крупные структурные изменения

Более крупным улучшением будет создание **графа знаний** поверх существующей структуры. Он может храниться как набор Markdown-таблиц и/или JSONL-файл, где явно указаны связи между терминами, организациями, источниками данных, информационными системами, отчётными формами и нормативными актами.

Второе крупное изменение — разработка **набора обучающих сценариев для ИИ-ассистента**. Каждый сценарий должен включать пользовательский вопрос, нужный маршрут по базе знаний, ожидаемый ответ, допустимые источники и критерии качества. Такие сценарии можно использовать для regression-тестов ассистента и оценки RAG.

Третье крупное изменение — оформление раздела `howto` как набора воспроизводимых практик. Для каждой инструкции стоит фиксировать входные данные, инструменты, шаги, ожидаемый результат, типовые ошибки и связанные страницы. Это сделает раздел пригодным не только для чтения, но и для автоматизированного выполнения ассистентом.

## 11. Рекомендуемый план работ

| Этап | Горизонт | Результат |
|---|---|---|
| Нормализация метаданных | 1–2 недели | Все страницы имеют `description`, контролируемые теги и тип контента. |
| Усиление README-хабов | 1–2 недели | Каждый раздел стал понятной точкой входа для пользователя и ассистента. |

## 12. Итог

Репозиторий уже имеет качественную основу для справочника по открытым государственным финансам и потенциально хорошо подходит для ИИ-ассистента. Его сильные стороны — предметная структура, Docusaurus-навигация, развитый Markdown-корпус, наличие `AGENTS.md` и почти полное покрытие frontmatter. Основные улучшения должны быть направлены на **обязательные описания страниц, сценарную навигацию, первоисточники, структурирование длинных материалов и создание машинно-читаемого слоя связей**. После этих доработок репозиторий сможет работать не только как сайт-справочник, но и как полноценная база знаний для RAG, обучения и оценки ИИ-ассистентов.
