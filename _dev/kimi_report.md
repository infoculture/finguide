

# Анализ репозитория opengovfinancesbook
## Предложения по улучшению содержания, навигации и пригодности для обучения ИИ-ассистента

**Дата анализа:** 2026-05-12
**Репозиторий:** https://github.com/infoculture/opengovfinancesbook
**Платформа:** Docusaurus 3 + Markdown

---

## 1. Общая оценка репозитория

Репозиторий представляет собой **высококачественную базу знаний** по государственным финансам России с продуманной канонической архитектурой (v2). Проект уже имеет отличную инфраструктуру для работы с ИИ-ассистентами:

- **AGENTS.md** — детальный контракт качества для ИИ-редакторов (Cursor и др.)
- **6 типов шаблонов карточек** с единой структурой frontmatter
- **Строгая проверка сборки** (битые ссылки прерывают build)
- **Канонические URL** для всех перекрестных ссылок
- **Карта данных** (data-map) с типовыми маршрутами пользователя
- **11 разделов** с четкой иерархией

**Оценка пригодности для ИИ/RAG: 7.5/10** — хорошая база с явными зонами роста.

### 2.1. Структурные улучшения сайдбара

| # | Проблема | Предложение | Приоритет |

| # | Предложение | Rationale |

| # | Предложение | Примечание |

## 4. Пригодность для обучения ИИ-ассистента — оценка и рекомендации

### 4.1. Что уже отлично работает для ИИ/RAG

| Компонент | Оценка | Комментарий |
|-----------|--------|-------------|
| AGENTS.md | 9/10 | Один из лучших примеров инструкций для ИИ-редакторов. Явные шаблоны, запреты на заглушки, требования к ссылкам |
| Структура frontmatter | 9/10 | title, description, tags, last_updated, source_url — полный набор для индексации |
| Семантические заголовки | 8/10 | `##` и `###` задают четкую иерархию, удобную для chunking'а |
| Канонические URL | 9/10 | `/glossary/kbk` вместо относительных путей — предсказуемость для ИИ |
| Карта данных (data-map) | 8/10 | 4 типовых маршрута — отличная навигационная эвристика для RAG |
| Перекрестные ссылки | 7/10 | Требование 2-3 конкретных ссылок в каждой карточке создает граф знаний |
| "Частые вопросы" в карточках | 7/10 | Q&A формат идеально подходит для fine-tuning на QA-пары |

### 4.2. Что нужно улучшить для ИИ-ассистента

| # | Проблема | Решение | Приоритет |
|---|----------|---------|-----------|

| 22 | **Нет набора валидационных Q&A пар** | Создать `tests/qa-validation.md` с 50-100 типовыми вопросами и каноническими ответами (ссылки на wiki) | Средний |

### 4.3. Рекомендуемая архитектура ИИ-ассистента на базе книги

```
Пользовательский запрос
       |
       v
+-------------------------------+
|  Router (намерение)           |
|  - определение термина        |
|  - поиск портала/данных       |
|  - поиск формы отчетности     |
|  - общий вопрос о бюджете     |
|  - практическая инструкция    |
+---------------+---------------+
                |
    +-----------+-----------+
    |                       |
    v                       v
+-----------+      +------------------+
| data-map  |      | Vector DB (RAG)  |
| (правила  |      | (wiki/ контент)  |
| маршрута) |      +--------+---------+
+-----+-----+               |
      |                     v
      |            +------------------+
      |            | Retriever        |
      |            | (top-k chunks    |
      |            |  с метаданными)  |
      |            +--------+---------+
      |                     |
      +----------+----------+
                 |
                 v
      +--------------------+
      | Prompt assembler   |
      | (AGENTS.md +       |
      |  шаблон карточки + |
      |  retrieved chunks) |
      +--------+-----------+
               |
               v
      +----------------+
      | LLM response   |
      | (с цитатами из |
      |  wiki/)        |
      +----------------+
```

### 4.4. Конкретные шаги для повышения пригодности к ИИ

**Фаза 1 (быстрые победы, 1-2 недели):**
1. Создать `KNOWLEDGE.md` — краткий конспект структуры книги для LLM
2. Добавить `scripts/export-to-rag.py` — экспорт контента в JSON с чанками
3. Заполнить 3-5 практических howto с кодом (Python)
4. Добавить теги сложности в frontmatter существующих карточек

**Фаза 2 (1-2 месяца):**
5. Создать `tests/qa-validation.md` с 50 QA-парами
6. Заполнить howto раздел (10-15 гайдов)
7. Добавить Mermaid-диаграммы в 10 ключевых карточек
8. Создать систему проверки живости внешних URL (CI)

**Фаза 3 (3-6 месяцев):**
9. Заполнить региональные и муниципальные источники данных
10. Добавить госкорпорации и АО с госучастием
11. Интегрировать векторный поиск на сайт
12. Добавить интерактивные элементы (чек-листы, фильтры)

## 5. Сравнение с лучшими практиками (benchmark)

| Критерий | opengovfinancesbook | Лучшие практики (Diataxis, Google TA) | Зазор |
|----------|-------------------|--------------------------------------|-------|
| Структура контента | Карточная система (6 типов) | Типы документации (tutorial, how-to, reference, explanation) | Нет tutorials для новичков |
| Шаблоны для редакторов | AGENTS.md + 6 шаблонов | Style guide + templates | На уровне лучших |
| Практические инструкции | Структура есть, контент минимален | Пошаговые гайды с кодом | Критический пробел |

| Поиск | Не настроен (предположительно) | Algolia / векторный поиск | Критический пробел |
| Визуализации | Нет | Mermaid-диаграммы, инфографика | Значительный пробел |

## 6. Итоговые рекомендации (приоритизированные)

### Must-have (делать немедленно)

1. **Заполнить howto раздел** — 3-5 практических гайдов с кодом (Python/Excel). Это главный пробел для обучения ИИ и пользователей.
2. **Создать `KNOWLEDGE.md`** — компактный файл для подключения к LLM: структура, правила, примеры запросов.
3. **Включить Algolia DocSearch** — поиск по 100+ страницам критически необходим.
4. **Добавить breadcrumbs** в Docusaurus для сохранения контекста навигации.

### Should-have (1-2 месяца)

5. Создать `scripts/export-to-rag.py` для экспорта в векторные БД.
6. Сгруппировать нормативную базу (legal/) по темам в сайдбаре.
7. Добавить `getting-started` страницу для новых пользователей.
8. Создать 50 QA-пар для валидации ответов ИИ.
9. Добавить Mermaid-диаграммы в ключевые карточки.
10. Начать наполнение региональных источников данных (10-15 крупнейших субъектов).

### Nice-to-have (3-6 месяцев)

11. Векторный поиск на сайте (не только Algolia).
12. JSON-LD разметка для структурированных данных.
13. Интерактивные чек-листы для практических задач.
14. Госкорпорации и АО с госучастием в отчетности.
15. Система проверки живости внешних URL в CI.
