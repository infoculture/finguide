# Анализ репозитория finguide: рекомендации по улучшению

## Обзор проекта

Репозиторий [infoculture/finguide](https://github.com/infoculture/finguide) — это структурированная база знаний «Открытые государственные финансы России», собираемая на Docusaurus 3 и публикуемая на [GitHub Pages](https://infoculture.github.io/finguide/). Проект объединяет описание бюджетной системы РФ, государственных информационных систем, источников открытых данных и практических инструкций. Явно заявленная целевая аудитория включает аналитиков, журналистов, разработчиков, а также ИИ-системы, использующие базу знаний как источник для RAG.

### Текущая структура

Wiki-контент организован в 11 тематических разделов:

| Раздел | Путь | Статус наполнения |
|--------|------|-------------------|
| Введение | `wiki/intro/` | Хорошо заполнен (8 файлов) |
| Глоссарий | `wiki/glossary/` | Хорошо заполнен (8 подпапок, 30+ карточек) |
| Организации | `wiki/organizations/` | Хорошо заполнен (15+ карточек) |
| Информационные системы | `wiki/information-systems/` | Хорошо заполнен (федеральные, 25+ карточек; региональные и гражданские — минимально) |
| Бюджетная система | `wiki/budget-system/` | Есть |
| Бюджетная классификация | `wiki/budget-classification/` | Есть |
| Источники данных | `wiki/data-sources/` | Федеральные — хорошо; региональные (3 файла), международные (2 файла), гражданские — слабо |
| How-to | `wiki/howto/` | Частично: крупные страницы и стабы |
| Нормативная база | `wiki/legal/` | Есть |
| Бюджетная отчётность | `wiki/reporting/` | Есть |
| Справочники | `wiki/reference/` | 13 файлов, хороший охват |

## 1. Проблемы содержания

### 1.1 Стабы в разделе How-to

Несколько страниц в `wiki/howto/analysis/` представляют собой минимальные заготовки (стабы) объёмом около 1–1,5 КБ с нерабочими примерами кода. Страница `debt-analysis.md`, например, содержит блок Python с полностью закомментированным кодом и отсылает к разделам «Информационные системы» и «Источники данных» без конкретных ссылок. Аналогично выглядят `gov-programs.md`, `kbc-decoding.md`, `procurement-analysis.md`, `subsidy-tracking.md`, `time-series.md` — каждая из них предположительно является скелетом, ожидающим реального наполнения.

**Рекомендации:**
- Каждый how-to должен содержать минимум один рабочий воспроизводимый пример (реальные URL, команды curl/Python с нередактируемым кодом или явным `# TODO: заменить URL`).
- До заполнения стабы следует либо пометить `draft: true` в frontmatter (чтобы они не попадали в индекс сайта), либо временно убрать из `sidebars.ts`.
- Хорошие примеры уже есть в `budget-execution.md` (~9,8 КБ), `ngo-funding.md` (~13,8 КБ) и `regional-budget-analysis.md` (~15 КБ) — их структура должна стать стандартом.

### 1.2 Незаполненные региональные и гражданские источники

В `wiki/data-sources/regional/` всего 3 содержательных файла (consolidated-budgets, municipal-budgets, regional-portals), а в `wiki/data-sources/international/` — только 2 карточки (CIS Stat и Всемирный банк). Раздел `wiki/data-sources/civil/` также содержит минимум материалов. При этом `wiki/intro/wiki-filling-plan.md` подробно описывает именно это как приоритет разделов 1–3 плана наполнения.

**Рекомендации:**
- Добавить карточки по крупнейшим региональным порталам: типовая страница «как найти региональный источник» с таблицей «субъект — URL — форматы — ограничения».
- Добавить международные источники: IMF GFS, OECD Public Finance, UN Stats — с блоком «связь с российскими данными».
- Для гражданских источников добавить явную страницу с критериями включения в wiki (официальный источник vs переупаковка, лицензия данных, дата последнего обновления).

### 1.3 Отсутствие карточек госкорпораций и ПАО с госучастием

Раздел `wiki/organizations/` охватывает Минфин, Казначейство, ЦБ, ФНС, ФТС, СФР, ФФОМС, Росстат, ФАС, Счётную палату, Госдуму и Совфед, но не содержит карточек Росатома, РЖД, Сбербанка, ВТБ, Газпрома и других крупных организаций с государственным участием. Их отчётность и источники данных — признанный пробел, явно зафиксированный в `wiki-filling-plan.md` как приоритет #1.

**Рекомендации:**
- Создать группу карточек «Организации с госучастием» в `wiki/organizations/` (госкорпорации, ППК, АО с долей государства >50%), используя существующий шаблон.
- Добавить соответствующие карточки источников данных (`wiki/data-sources/federal/`) и нормативных документов (`wiki/legal/`).

### 1.4 Неравномерная детализация карточек в разных разделах

Карточки глоссария в подпапке `budget-system` достигают 13 КБ (например, `treasury-execution.md`, `budget-process.md`), тогда как часть карточек в других подпапках значительно меньше и возможно не следует всем разделам шаблона. `AGENTS.md` требует конкретных перекрёстных ссылок (не заглушек вида «см. раздел глоссария»), однако это правило может выполняться неравномерно.

**Рекомендации:**
- Провести аудит карточек с размером менее 3 КБ на предмет соответствия шаблону (наличие разделов «Связанные термины», «Где встречается в данных», «Частые вопросы»).
- Добавить в CI автоматическую проверку frontmatter: обязательные поля `slug`, `last_updated`, `tags` должны присутствовать во всех карточках глоссария, ИС, организаций и источников данных.

## 2. Проблемы навигации

### 2.1 Отсутствие машиночитаемого индекса сущностей

В репозитории нет единого каталога (JSON или CSV), перечисляющего все карточки по типам (термин / ИС / организация / источник) с их slug, тегами и датой обновления. Это затрудняет как навигацию пользователя по тегам, так и RAG-индексацию.

**Рекомендации:**
- Добавить генерацию `index.json` как шаг npm-сборки или отдельный скрипт: тип карточки, slug, теги, `last_updated`.
- Либо использовать Docusaurus plugin для генерации поискового индекса (algolia/local search) с фасетами по типам сущностей.

### 2.2 Теги не связаны с навигационными страницами

Таксономия тегов описана в `wiki/reference/tags-taxonomy.md`, но в Docusaurus теги по умолчанию не формируют автоматических навигационных страниц для docs (только для blog). Читатель не может нажать на тег и увидеть все карточки с ним.

**Рекомендации:**
- Рассмотреть плагин `@docusaurus/plugin-content-pages` с кастомной страницей тегов или использовать `@docusaurus/plugin-content-docs` с включённым `tagsBasePath`.
- Временное решение: добавить в `wiki/reference/` вручную ведущиеся страницы-индексы по ключевым тегам (`rag`, `federal`, `regional`, `howto`, `legal`).

### 2.3 Карта данных не покрывает новые материалы

`wiki/intro/data-map.md` — полезный навигационный инструмент, описывающий маршруты «вопрос → глоссарий → ИС → источники данных → отчётность». Однако после добавления новых карточек (что планируется по всем разделам плана) карта потребует обновления. Сейчас нет механизма, напоминающего о необходимости её обновлять.

**Рекомендации:**
- Добавить `data-map.md` в обязательный чеклист PR для изменений, затрагивающих навигационные маршруты.
- Рассмотреть автогенерацию части карты (например, список ИС по категориям) из frontmatter файлов через скрипт.

### 2.4 Дублирование навигационных конфигураций (`.js` и `.ts`)

В корне репозитория сосуществуют `docusaurus.config.js` и `docusaurus.config.ts`, а также `sidebars.js` и `sidebars.ts`. Это потенциальный источник рассинхронизации конфигурации.

**Рекомендации:**
- Удалить `.js`-версии файлов конфигурации, оставив только `.ts`.
- Добавить явное примечание в CONTRIBUTING.md о том, что правки вносятся только в `.ts`-файлы.

### 2.5 Раздел `wiki/reference/` смешивает разнородные материалы

В `reference/` находятся одновременно: таксономия тегов, справочники кодов (ОКАТО/ОКТМО), принципы открытых данных (30 КБ), журнал обновлений, внешние ссылки и международные стандарты. Раздел выполняет роль «прочего» и может затруднять поиск.

**Рекомендации:**
- Крупный файл `principles.md` (30 КБ) вынести в самостоятельный раздел или в `intro/`.
- Справочники кодов (ОКАТО, коды организаций, идентификаторы открытых данных) сгруппировать в подпапку `reference/classifiers/`.
- `links.md` и `openprojects.md` объединить или перенести в раздел гражданских источников.

## 3. Пригодность для обучения и использования ИИ-ассистента (RAG)

### 3.1 Сильные стороны существующей структуры

Репозиторий изначально проектировался с учётом RAG: страница `about.md` прямо называет ИИ-системы одной из целевых аудиторий, `how-to-use.md` описывает принцип «один файл — одна основная сущность» и рекомендует семантические заголовки как якоря для RAG, а `AGENTS.md` содержит подробные инструкции для агентов по созданию и обновлению карточек. Наличие `draft: true` в frontmatter шаблонов предотвращает попадание пустых страниц в production-сборку.

### 3.2 Проблемы для RAG-индексации

**Отсутствие поля `description` в ряде карточек.** Поле `description` в frontmatter — критически важно для embedding-моделей при сжатии контекста: оно позволяет включать страницу в топ-k результатов даже при коротком запросе. Шаблон глоссария помечает `description` как рекомендованное, но не обязательное поле.

**Нет явной маркировки типа сущности.** Разные типы карточек (термин, организация, ИС, источник) имеют поле `tags`, но не имеют единого поля `entity_type`. Это затрудняет фильтрацию при построении контекста для ответов разного типа («дай определение» vs «найди источник данных»).

**Секции FAQ не стандартизированы по формату вопрос/ответ.** В одних карточках FAQ оформлен через `**В:** / **О:**`, в других — произвольно. Для оптимального RAG предпочтителен строгий Q&A-формат с однородной разметкой.

**Внешние URL в поле `source_url` и в теле карточек могут устаревать.** Нет автоматической проверки живости внешних ссылок в CI.

### 3.3 Рекомендации по улучшению для RAG

- **Сделать `description` обязательным** во всех шаблонах карточек (не более 2-3 предложений, включающих ключевые термины сущности).
- **Добавить поле `entity_type`** в frontmatter: значения `glossary`, `organization`, `information-system`, `data-source`, `reporting-form`, `legal-document`, `howto`. Это позволит фильтровать чанки при построении контекста.
- **Унифицировать формат FAQ** по всем карточкам: `**В:** ... \n**О:** ...` (как указано в шаблоне глоссария) — внести явное требование в `AGENTS.md`.
- **Добавить в CI проверку живости внешних ссылок** (например, через `lychee` или `linkcheck` в GitHub Actions) с периодичностью раз в неделю.
- **Создать специальный файл `llms.txt`** в корне репозитория (по аналогии с [llms.txt standard](https://llmstxt.org/)) с кратким описанием структуры базы знаний и рекомендуемыми точками входа для ИИ-агентов.
- **Структурировать перекрёстные ссылки машиночитаемо**: рассмотреть добавление поля `related_pages` (список slug) в frontmatter карточек помимо prose-ссылок в теле — для построения графа знаний.
- **Добавить страницу `wiki/intro/rag-guide.md`** — явный гайд для разработчиков RAG-пайплайнов: какие чанкинг-стратегии подходят, какие поля frontmatter использовать для метаданных, какие разделы индексировать в первую очередь.

### 3.4 Оценка пригодности для дообучения (fine-tuning)

Для использования контента как обучающих данных для LLM (не RAG, а fine-tuning) текущий корпус имеет следующие характеристики:

| Аспект | Оценка | Комментарий |
|--------|--------|-------------|
| Структурированность | Высокая | Единые шаблоны карточек, frontmatter |
| Фактическая точность | Высокая | Ссылки на НПА, официальные URL |
| Полнота предметной области | Средняя | Региональные/гражданские источники слабо покрыты |
| Q&A-пары | Средняя | Есть, но неравномерно по разделам |
| Объём | Средний | Достаточен для domain-specific LoRA, недостаточен для full fine-tuning |
| Языковая однородность | Высокая | Только русский язык |

Для fine-tuning рекомендуется дополнительно создать датасет Q&A-пар из существующих секций «Частые вопросы» в формате JSONL, что даст дополнительный обучающий сигнал на понимание предметной области.

## 4. Технические и инфраструктурные улучшения

### 4.1 CI/CD

Текущий CONTRIBUTING.md требует `npm run build` перед PR и обещает проверку битых ссылок. Рекомендуется расширить CI:

- Добавить `npm run typecheck` как обязательный шаг GitHub Actions (сейчас упомянут, но неизвестно, включён ли в workflow).
- Добавить lint frontmatter (обязательные поля, формат slug, `last_updated` не старше 1 года для карточек источников).
- Добавить еженедельную проверку внешних ссылок с автоматическим созданием Issue при обнаружении битых URL.

### 4.2 Шаблоны и дублирование конфигурации

Файлы конфигурации в двух форматах `.js` и `.ts` — это технический долг. После перехода на Docusaurus 3 с TypeScript-конфигурацией `.js`-файлы стали избыточными.

### 4.3 Отсутствие Issues для отслеживания прогресса

Репозиторий не имеет открытых Issues, хотя `wiki-filling-plan.md` содержит 9 разделов приоритетных задач. Это затрудняет координацию и прозрачность для внешних участников.

**Рекомендации:**
- Перенести пункты `wiki-filling-plan.md` в GitHub Issues с метками `content`, `priority-1`, `priority-2` и т.д.
- Создать GitHub Project для отслеживания прогресса наполнения по разделам.
- Добавить GitHub Issue Templates для типовых операций: «добавить карточку источника», «обновить карточку ИС», «исправить битую ссылку».

## 5. Приоритизированный план действий

### Приоритет 1 — Немедленно (улучшают качество существующего контента)

1. Пометить `draft: true` все стабы в `wiki/howto/analysis/` (5–6 файлов менее 2 КБ).
2. Удалить дублирующие `.js`-конфиги, оставив только `.ts`.
3. Сделать `description` обязательным полем во всех шаблонах карточек (`AGENTS.md`).
4. Добавить поле `entity_type` в frontmatter всех шаблонов карточек.

### Приоритет 2 — Среднесрочно (расширение покрытия)

5. Создать хотя бы 3–5 карточек региональных источников данных (крупнейшие субъекты или методическая страница).
6. Добавить карточки 2–3 международных источников (IMF GFS, OECD, UN Stats).
7. Создать группу карточек госкорпораций в `wiki/organizations/`.
8. Наполнить 3–4 приоритетных стаба how-to рабочими примерами кода.

### Приоритет 3 — Долгосрочно (инфраструктура и навигация)

9. Создать `llms.txt` в корне репозитория.
10. Добавить автоматическую проверку живости внешних ссылок в CI.
11. Перенести задачи из `wiki-filling-plan.md` в GitHub Issues.
12. Добавить страницу `wiki/intro/rag-guide.md`.
13. Рассмотреть поддержку тегов-страниц в Docusaurus или ручной индекс по тегам.
